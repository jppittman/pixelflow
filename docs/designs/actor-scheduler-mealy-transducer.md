## 1. Overview

An actor is a **Mealy machine**: `δ : State × Input → State × Output`. A handler takes
`&mut self` (the state transition) and the incoming message (the input symbol), and it
**returns** the messages it wants to emit (the output word). It does not call `send`. It
does not block. It does not suspend. It runs to completion and returns.

```rust
fn handle_control(&mut self, msg: C) -> Result<Out, HandlerError>;
//  &mut self = S → S'      msg = input      Ok(out) = output word     Err = failed step
```

The scheduler owns everything the handler used to: it takes the returned output, routes it,
applies backpressure, and decides when the actor runs again. That single inversion — **effects
are return values, not calls** — is the whole design. Everything else falls out of it.

### 1.1 Why the Kubernetes framing died

The prior design (`actor-scheduler-supervisor-migration.md`) modelled the system as a cluster:
Node = core, Kubelet = thread, Pod = actor group, ClusterIP = stable address. It was a
*supervision* metaphor wearing a *scheduling* costume, and it explicitly punted on the thing
that actually mattered (`Non-Goals`: "Preemptive scheduling within the kubelet — cooperative
only"). It also welded *actor* to *OS thread*, so adding an actor cost a thread — the opposite
of the goal. The kubelet that got built was a restart babysitter, and `poll_once` — the
cooperative step it needed — leaked the scheduling strategy into the actor's public API.

The replacement is not a smaller cluster. It is a different primitive. Supervision becomes a
plain concern (a supervisor waits on exits and restarts), and scheduling becomes a property of
the primitive itself. No pods, no clusterIPs, no kubelet in the common path.

### 1.2 Goals

- An actor is as cheap to add as a `Mutex` — an allocation, never a thread.
- Separate the **logical** unit (the actor) from the **hardware** unit (thread / core / process).
- No `async`, no coroutines, no generated state machine, no yield macro.
- Cooperative multiplexing where the *actor step* is the scheduling quantum.
- Backpressure and yielding are the scheduler's job, invisible to the handler author.

### 1.3 Non-Goals

- Preemption *inside* a handler (a handler is run-to-completion; anything that must be preempted
  gets its own thread or process — see §5).
- Dynamic 1000-actor swarms. This is a concurrency *primitive*, not a runtime for a fleet.
- Changing the three public priority lanes (Control > Management > Data) or the System lane.

## 2. The primitive

### 2.1 State lives in `self`; the handler is pure between inputs

Because the handler returns its effects instead of performing them, it never has a live stack to
suspend. Everything that must survive from one input to the next is already in `self`. There is
no continuation to capture, no program counter, no boxed future. A handler is a pure function
`(state, input) → (state, output)` with the state mutation done in place.

It is **Mealy, not Moore**: the output is a function of state *and* input (you echo the byte you
just received). Moore would only see state. Mealy is also the tighter encoding — fewer states
for the same behavior.

### 2.2 Output is a struct of typed ports (fan-out)

Fan-out targets have *different* message types (`app → engine` is `EngineMsg`; `app → write` is
`PtyMsg`). A `Vec<Msg>` or a union enum would erase that. The topology is static, so the output
is a struct with one optional, statically-typed slot per downstream port:

```rust
struct AppOut {
    engine: Option<EngineMsg>,   // fire the engine port, or not
    write:  Option<PtyMsg>,      // different type, still checked
}

// fan-out: Ok(AppOut { engine: Some(Render), write: Some(Echo(b)) })
// silence: Ok(AppOut::default())     // all ports None
```

This is the hardware Mealy picture literally: N output lines, each transition asserts a subset.
Zero-alloc (a fixed struct), each port type-correct to its target, generated by the macro from
the actor's declared outputs. Each port's message variant already encodes its priority lane, so
routing carries the lane for free.

### 2.3 A port addressed to yourself is the yield

To continue working without monopolising the worker, emit a message to your **own** inbox and
return. The scheduler runs everyone else, comes back, feeds you your own message, and you resume
on the next step. That is the "yield on send" this design chased for weeks — with no coroutine,
no `async`, no `pc`. It is a self-edge in the machine. Cooperative yield is an ordinary output.

Open question (§9): whether the self-port is a *named verb* in the macro or just "a port that
happens to point home."

## 3. The scheduler

The scheduler is the only thing that sends. On each step it:

1. Calls one handler with one input, gets back `Ok(Out)` or `Err(e)`.
2. For each `Some` port in `Out`, stages the message to that target's ring.
3. If a target ring is full, flushes the ports it can and **holds the remainder** — a parked
   actor's outbox is just the `Out` struct with already-flushed ports zeroed. Ports are
   independent, so partial flush is trivial.
4. Does **not** call that actor's next handler until its outbox has drained (correct
   backpressure: an actor cannot outrun its slowest consumer; it just parks).
5. On `Err(e)`, runs the existing error/restart path. `Err` carries no output — "emit *and*
   warn" is a message variant, not an error. The mutual exclusion keeps a step legible.

The resume trigger is the existing wake ("target ring not-full" → wake the parked sender), which
is a doorbell event on the **System lane** — the same lane that carries Wake/Shutdown, and the
natural home for a reactor (an `epoll_wait` fed by an in-process `eventfd`, IO fds, timers, and
— across processes — a `signalfd`). See §5.

## 4. Why there is no macro, no async, no coroutine

**The type shape is the style guide.** Because output is a return value, the only place a
handler can emit is its end. The instant you want to send, the handler is over and control is
back at the scheduler. A sprawling stateful handler that wants to fire mid-computation has
nowhere to put that fire except the return — which pressures it into smaller maps. The
architecture does not *permit* the sprawling handler; it makes the tight one the only pleasant
one. Frequent yield is not a feature that was added — it is the residue of "effects are return
values." A macro would only re-enforce what the signature already refuses.

`poll_once` is removed. The stepping it exposed still exists — it is `handle`, driven by the
scheduler — but it is no longer in anyone's public API. The `poll_once` leak was about *who
drives the stepping* (the caller, publicly), not that stepping exists. Runtime-driven stepping
behind a straight-line authoring model is the fix, not the leak.

## 5. Placement: logical actor vs hardware

The actor is logical. Where it runs is a separate, one-line decision on the *same* trait:

| Placement | Mechanism | Use for |
|-----------|-----------|---------|
| **Multiplexed** (default) | Cooperative on a shared worker; the actor step is the quantum | The common case; cheap, no thread |
| **Dedicated thread** | Own OS thread, kernel-preempted | Blocking-in-`park` (epoll, platform loop) or latency-critical |
| **Own process** | Separate address space, kernel-preempted, fault-isolated, `SIGSTOP`-pausable | CPU-heavy or untrusted (e.g. a compiler actor) |

Rules that fall out:
- A multiplexed actor **may not block** — not in a handler, not in `park`. Blocking on the OS
  forces the dedicated tier. (This is why `park` stays: it is the OS-bridge hook for that tier.)
- Preemption granularity is the process/thread, never the multiplexed green actor. Anything that
  needs real preemption gets its own thread or process. A signal nudges a worker to yield at
  step boundaries; it cannot interrupt a handler mid-body (nor should it need to).

### 5.1 Cross-process transport

Worker processes are the isolation/preemption grain. Communication stays fast because the SPSC
ring is already pointer-free (`[MaybeUninit<T>; N]` + atomic head/tail), so it drops into a
shared segment unchanged. The rules of shared memory:

- **A pointer in shared memory is an offset.** No `Box`/`Vec`/`Arc`/`String` across the boundary
  — they store the creating process's addresses. Cross-process references are relative
  (`Shared<T>` = segment + offset, resolved per-process).
- **`ShmSafe`** marks pointer-free / relocatable types. It bounds cross-process channels and the
  shm allocator, and it is the natural extension of the existing "Copy iff ZST" rule. Misuse is
  a compile error, not a wild write. Intra-process actors are unconstrained; only process
  boundaries pay the `ShmSafe` tax, and the compiler enforces it. Library consumers rarely touch
  it (a macro allocates into the shared arena for the rare payload that outgrows a POD).
- Shared memory is a **deliberate aperture** in the process isolation: isolate execution, share
  specific typed regions (the rings, the framebuffer), keep them minimal.

### 5.2 The System lane is the reactor

Wake, Shutdown, IO readiness, timers, and cross-process signals are all System-priority events
sharing one park point (`epoll_wait` / `kqueue`). `signalfd` folds a peer process's signal into
that set. Priority ordering is unaffected — it is applied when lanes are drained, not at the
reactor. Blocking IO lives on dedicated-thread actors that turn readiness into messages; a
Tier-0 IO actor forwarding a `DataReady` message *is* a Waker in disguise.

## 6. Determinism and testing

`(state, input) → (state, output)` is a pure function, so an actor is table-testable: feed a
symbol, assert the output word. No scheduler in the loop, no mocked sends. For a terminal
emulator this is a large win — the ANSI parser and the terminal state machine become property
tests over transition tables.

## 7. What changes in the code

| Change | Where |
|--------|-------|
| `handle_*` returns `Result<Out, HandlerError>` (was `Result<(), _>`); the typed `Out` struct is generated | `actor-scheduler/src/lib.rs`, macros |
| Typed-handle macro retargets generated calls from *ring send* to *output-port assignment* | `actor-scheduler-macros` |
| Scheduler owns staging + partial flush + park-on-full; wake on ring-not-full | `actor-scheduler/src/lib.rs` |
| `poll_once` removed from the public API | `actor-scheduler/src/lib.rs` |
| Supervisor (restart/frequency-gate) survives as a plain concern, decoupled from scheduling | `actor-scheduler/src/kubelet.rs` (to be de-Kubernetes-ified) |

Recommended first step: prove the runtime **before** touching the macro. Hand-write one actor in
the `handle → Out struct` form that fans out and parks on a full downstream ring, and confirm the
scheduler stages, partial-flushes, and resumes it correctly. Then generate that shape.

## 8. Paths not taken (the subtractive arc)

This design was reached by deletion. Recorded so it is not re-derived:

- **Kubernetes / kubelet / pods / ClusterIP** — a supervision metaphor masquerading as
  scheduling; welded actor to thread. Dropped (§1.1).
- **`poll_once` as public API** — leaked the scheduling strategy to the caller. Removed (§4).
- **Green-thread executor with dynamic spawn / work-stealing** — machinery for a 1000-actor
  fleet that is not the goal. The goal is a primitive.
- **Hidden `async` / coroutine macro (pc-in-`self`)** — both exist only to suspend a live stack
  mid-handler. Returning effects means there is no stack to suspend, so both are unnecessary.
- **`&mut Outbox` sink** — a returned value into a caller buffer is the same thing without the
  threaded parameter; the return reads as more transparent (effects-as-values).
- **A yield macro** — the self-port already yields; the type shape already forces frequent
  yielding. Nothing to enforce.

Each removal made the primitive more opinionated, not less capable. That is the tell.

## 9. Open questions

- [ ] **Self-port as a named verb?** First-class `yield`/continuation syntax in the macro, or
  just "a port that points home." Vocabulary vs. emergent trick.
- [ ] **Burst cardinality.** A port fires at most once per step; bursts batch into one message
  (`ScreenOps(batch)`). Confirm no actor needs a port to emit a *variable number of distinct*
  messages per input (which would make that port carry a collection).
- [ ] **`park` for the time-driven tier.** Timer ticks are just input symbols; confirm `park` on
  the dedicated tier is the right place to synthesise them.
- [ ] **De-Kubernetes-ify the supervisor.** What the restart/frequency-gate logic is called and
  where it lives once "kubelet/pod" vocabulary is gone.
